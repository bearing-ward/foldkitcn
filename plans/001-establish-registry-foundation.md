## Why this matters

This repo is intended to become a Foldkit-native component registry with high-fidelity ports of Base UI and shadcn components, plus docs, examples, themes, blocks, charts, and later custom/private registries. The current repo is still a starter Foldkit app, so a direct component port would force critical conventions to live only in conversation and one-off code. This plan establishes the durable architecture, schemas, source/generated registry split, validation checks, parity harness skeleton, and repo-local component-planning skill needed before any component can be responsibly imported.

The first real component proving batch after this plan should be `base-ui/button` plus `shadcn/button`, generated by the new skill as its own improve plan. Do not implement Button in this plan.

## Architecture decisions to encode

Create `docs/decisions/0001-foldkit-registry-architecture.md` as an ADR-style decision record. It must include these decisions in durable form:

- This is a Foldkit-native registry. Origin APIs are reference material; public APIs must use Foldkit `Model`, `Message`, `init`, `update`, `view`, Submodel, OutMessage, commands, and `toView`/part-renderer patterns where appropriate.
- All boundary data is Effect Schema-backed: registry manifests, origin dossiers, dependency classifications, theme tokens, parity result records, docs data, component `Model`, `Message`, and route data. Types derive from schemas.
- The editable source of truth is `registry-src/<namespace>/<item>/`; generated/public registry output lives under `registry/`, `public/r/`, or `dist/` and is not hand-edited.
- Item IDs are path-like and match their folder path, such as `base-ui/button`, `shadcn/button`, `shadcn/blocks/sidebar-01`, `shadcn/charts/area`, `utils/cn`, `themes/default`, and `local/acme-widget`.
- Every upstream-derived item is pinned to immutable origin provenance: origin library, docs URL, source URL, local repo path, git ref/commit, source/docs/examples/tests inventory, and inventory hash.
- Source manifests are human-authored `item.json` files validated by shared Effect schemas. Generated hashes, computed dependency graphs, parity run results, drift reports, and docs URLs do not belong in `item.json`.
- Base UI namespace items are unstyled behavior primitives by default. Origin styled demos become executable examples layered on top.
- Shadcn namespace items are styled Foldkit wrappers/compositions that depend on local registry primitives instead of upstream Base UI/Radix React packages.
- Shadcn items that reference Base UI must depend on local `base-ui/*` items, not duplicate them and not import upstream packages.
- Themes are registry items/theme packs. Component folders declare consumed tokens; theme packs declare covered namespaces/items.
- Blocks are first-class registry items whose runtime requirements must be met locally. A block plan expands into a dependency-complete batch if required.
- Charts are gated on an explicit native chart foundation. Chart requests should be recognized but marked as needing the chart foundation until it exists.
- Live docs URLs are discovery inputs, not parity oracles. Parity compares local pinned-origin fixtures against local Foldkit implementations.
- React is allowed only in origin fixture infrastructure, never in installable Foldkit runtime source.
- Public installability is gated by accepted parity, schema validation, generated registry output, examples/docs data, resolvable dependencies, and no unaccepted required deviations.
- Lifecycle is multi-dimensional: implementation status, parity status, drift status, and availability. Do not collapse them into one overloaded enum.
- Upstream drift is visible but non-blocking by default because pinned components remain reproducible.
- The future playground exports drafts into the registry workflow; it must not mutate accepted registry items directly.
- `components.build` principles are advisory standards captured in docs/checklists, while local schemas/tests/parity checks enforce acceptance.
- The first proving batch after this plan is `base-ui/button` plus `shadcn/button`; it should be produced as a plan by the skill created here.
