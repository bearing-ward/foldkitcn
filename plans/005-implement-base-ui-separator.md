## Why this matters

Separator is a small but important primitive for proving that the registry can
ship non-interactive, unstyled Base UI behavior with high origin fidelity. It
also unlocks the shadcn separator wrapper in plan 006, which must compose the
local Foldkit primitive instead of importing Base UI React. The implementation
should preserve the origin role, orientation default, ARIA, data attributes, and
render replacement capability while using Foldkit `toView` instead of React
`render` or `asChild`.

## Steps

### Step 1: Create the Foldkit primitive

Create `src/registry/base-ui/separator/index.ts` using the structure from
`src/registry/base-ui/button/index.ts`.

Required source shape:

- Import `Schema as S` from `effect`.
- Import `type Attribute, type Html` and `html` from `foldkit/html`.
- Add section headers `// MODEL` and `// VIEW`.
- Define `SeparatorOrientation` as an Effect Schema union of exactly
  `'horizontal'` and `'vertical'`.
- Define `SeparatorOptions` as an Effect Schema struct with optional
  `orientation`.
- Export TypeScript types derived from the schemas with
  `typeof SeparatorOrientation.Type` and `typeof SeparatorOptions.Type`.
- Define `SeparatorAttributes<Message>` as a named part:
  `separator: ReadonlyArray<Attribute<Message>>`.
- Define `ViewConfig<Message> = SeparatorOptions & { toView: (...) => Html }`.
- In `view<Message>(config)`, bind `const h = html<Message>()` inside the
  function.
- Default `orientation` to `'horizontal'`.
- Call `toView` with attributes in this order:
  `h.Role('separator')`, `h.AriaOrientation(orientation)`,
  `h.DataAttribute('orientation', orientation)`.

The local API intentionally maps Base UI React `render` to Foldkit `toView`.
Do not add React-style `asChild`, `render`, refs, `className`, or `style` to the
Base UI primitive. Callers can add classes and alternate elements in `toView`.

**Verify**: `bun run typecheck` should at least compile the new source; if it
fails for unrelated pre-existing files, capture the error and continue only if
the separator source has no type errors.

### Step 2: Add semantic Scene tests

Create `src/registry/base-ui/separator/scene.test.ts` following
`src/registry/base-ui/button/scene.test.ts`.

Cover these cases:

- Default separator renders a visible element with role `separator`.
- Default separator has `aria-orientation="horizontal"` and
  `data-orientation="horizontal"`.
- Vertical separator has `aria-orientation="vertical"` and
  `data-orientation="vertical"`.
- `toView` can choose an alternate element while preserving the separator
  attributes. Use a non-interactive element; do not introduce messages or state
  that the primitive does not own.

Keep Foldkit conventions:

- Use `m()` for any test-only message schema if a Scene harness requires one.
- Use `const h = html<Message>()` inside view functions.
- Use `empty`, keyed wrappers, and `evo` only if the test app actually needs
  them; separator should stay stateless.

**Verify**: `bun run test -- src/registry/base-ui/separator/scene.test.ts`
should pass.
