## Anti-Patterns Verdict

**LLM assessment:** The docs do not look obviously AI-generated, but they do look mechanically generated and under-edited. Repeated uppercase eyebrows, identical component rows, huge fluid headings, uniform ruled sections, and registry-derived prose overwhelm the otherwise credible technical shell. The bordered brand mark with a broad `0 18px 45px` shadow is the exact ghost-card pattern the design context prohibits.

**Deterministic scan:** The detector returned 18 advisory findings: 13 font-size, 2 radius, and 3 color findings. Fifteen are in `src/styles.css`; three are within origin-style registry examples. The strongest actionable evidence is docs-shell type-ramp drift across near-duplicate sizes (`0.88rem`, `0.9rem`, `0.92rem`, `0.95rem`, `1.05rem`). Preview-scoped colors and radii are intentional origin-fidelity exceptions, and the `0.6)` radius hit is a parser false positive.

**Visual overlays:** No user-visible overlay was created because the connected browser runtime reported `No browser is available`. The live URL returned HTTP 200. Independent visual review used a fresh headless browser across representative routes instead.

## Overall Impression

The documentation is credible, functional, and honest about component status. Its largest opportunity is to become an evaluation-and-installation tool rather than a rendered inventory: preview, Foldkit difference, readiness, and install confidence should lead; catalog exhaustiveness and architecture detail should follow.

## What's Working

1. The shell and origin-compatible preview themes are visibly separate, preserving both Foldkit identity and component parity.
2. Availability, preview readiness, missing documentation, source links, and lifecycle language avoid presenting unfinished work as installable.
3. Search, sticky navigation, install tabs, copy actions, live previews, source snippets, and responsive navigation use familiar developer-documentation affordances.

## Priority Issues

### [P1] Discovery is an exhaustive inventory, not a navigable catalog

**Why it matters:** Existing Foldkit developers arrive with a UI need, not necessarily a component name. Approximately 100 sidebar links and long repeated index rows make breadth feel like complexity.

**Fix:** Make search the primary discovery control; collapse namespaces; introduce task-oriented groups or filters; keep alphabetical browse secondary; reduce index rows to name, purpose, availability, and docs status.

**Suggested command:** `$impeccable distill docs`

### [P1] Component pages answer architecture questions before evaluation questions

**Why it matters:** Developers must first decide whether a component looks and behaves correctly, then install it confidently. Leading with composition internals reverses that workflow.

**Fix:** Place one representative live preview directly below the header. Add a compact trust/status rail for installability, docs readiness, origin, parity evidence, and namespace. Follow with installation. Move composition into an implementation section and summarize Foldkit differences near the top.

**Suggested command:** `$impeccable layout docs`

### [P1] Mobile becomes an extremely long undifferentiated scroll

**Why it matters:** Removing both sidebar and TOC eliminates section context, while repeated large preview/code blocks make API and Foldkit differences expensive to reach.

**Fix:** Add a compact sticky “On this page” disclosure, collapse secondary examples, and ensure install commands communicate horizontal scrolling or reflow safely.

**Suggested command:** `$impeccable adapt docs`

### [P2] Visual hierarchy is oversized and too uniform for a product tool

**Why it matters:** The 88px desktop title dominates actionable information. Repeated dividers, bordered containers, and many micro font sizes flatten priority below it.

**Fix:** Replace the fluid route title with a fixed product-scale heading around 40–48px desktop and 32–36px mobile. Consolidate the type ramp and prioritize status, preview, and installation ahead of implementation detail.

**Suggested command:** `$impeccable typeset docs`

### [P2] Foldkit's differentiator is described but not designed into the first screen

**Why it matters:** “shadcn ported to Foldkit” is the positioning, but visible proof of the translation arrives late. The experience risks reading as a shadcn registry with a Foldkit label.

**Fix:** Add a consistent Foldkit-port summary showing state ownership, emitted messages, underlying primitive, source origin, and documented differences. Replace generated artifact count as the homepage's leading proof.

**Suggested command:** `$impeccable clarify docs`

## Persona Red Flags

**Alex — existing Foldkit power user:** Known-URL installation is efficient, but browsing 100 entries lacks task filters or compact expert navigation. Long example sequences separate install/source tasks from API, quality, and Foldkit differences.

**Sam — keyboard or screen-reader-dependent developer:** Semantics are generally strong, but the first reading path traverses a very large navigation, no skip-to-content control was visible, mobile removes the TOC, focus uses a browser-default outline rather than the documented vocabulary, and visible copy confirmation remains weak.

**Casey — distracted mobile developer:** Long pages provide no persistent section context, repeated previews delay API/differences, and the install command was visually clipped at 390px without an obvious scroll cue.

## Minor Observations

- The static “Composes” card uses more space than its information deserves.
- “Front door for installable component registry artifacts” is system-centered compared with “shadcn components, ported to Foldkit.”
- A copyable registry URL contains `<owner>` and `<repo>` placeholders and can look ready before substitution.
- Uppercase tracking is repeated across eyebrows, search, sidebar headings, badges, metadata, and TOC labels.
- Human-facing TOC labels expose source-style example names such as `ButtonDefault`.
- The semantic error color at `src/styles.css:1084` is absent from the captured design tokens.

## Questions to Consider

- What if every component page answered “Should I use this?” before “How is it composed?”
- What is the smallest visible proof that this is a Foldkit port rather than a renamed shadcn component?
- Do fourteen Button examples need equal visibility, or should three lead and the rest live behind a variant index?
- If search is the fastest route through 100 components, why does the exhaustive list have more visual authority?
- Could availability, docs status, origin parity, and Foldkit differences become one trust contract beside installation?
